# Fase 6 Dashboard Design Brief

Tanggal: 2026-04-29

## Objective

Redesign the static dashboard into a modern, interactive, responsive cybersecurity/IoT forensics analytics website. The dashboard must remain artifact-driven: every metric/chart must be loaded from `dashboard/data/dashboard-data.json`, which is generated by `scripts/generate_dashboard_data.py`.

## Design Direction

UI/UX Pro Max recommendation:

- Pattern: Real-Time / Operations Landing
- Style: OLED dark mode / cybersecurity SOC dashboard
- Typography: Fira Sans + Fira Code
- Effects: minimal glow, subtle glass panels, visible focus, smooth 150–300ms hover transitions
- Responsive targets: 375px, 768px, 1024px, 1440px
- Accessibility: visible text values, not color-only, keyboard focus states, reduced-motion support

## Layout Proposal

1. Sticky top navigation
   - Project logo/mark: IoT DoS Forensics IDS
   - Anchor links: Overview, Dataset, Modeling, Forensics, Advanced, Evidence
   - Theme toggle optional, default dark.

2. Hero / command center
   - Large title and concise thesis.
   - Status pill from project status.
   - Highlight chips: BoT-IoT, DoS/DDoS, Digital Forensics, UNSIL Semester 6.
   - Right-side “threat monitor” visual card with live-style metrics derived from data.

3. Executive KPI bento
   - Rows audited
   - Normal rows train/test
   - Completed baseline runs
   - Completed advanced runs
   - Track A improvement delta
   - Top SHAP/forensic feature group

4. Dataset & imbalance panel
   - Donut/bar chart for class distribution.
   - Warning/limitation callout: normal class very small, accuracy not main claim.

5. Modeling comparison panel
   - Interactive tabs/filter for Baseline vs Advanced.
   - Bar chart: macro F1 by model/track.
   - Cards: best baseline, best advanced, Track A highlight.

6. Confusion matrix / error panel
   - Select track/model from available confusion rows.
   - Render custom 2x2 matrix with TN/FP/FN/TP.
   - Explain FP/FN in Indonesian.

7. Forensic evidence panel
   - Feature importance ranking bars.
   - SHAP ranking bars.
   - Interpretability notes tied to IoT network traffic.

8. Methodology/evidence timeline
   - Fase 1–6A progress timeline.
   - Links to review files and reports.

9. Limitations panel
   - Normal class tiny.
   - Controlled subset can overstate performance.
   - SHAP is sampled.
   - Split-similarity risk.

## Recommended Libraries

Static CDN is acceptable because dashboard is GitHub Pages-based:

- Chart.js for charts.
- Lucide icons for consistent SVG icons.
- No build system required.
- Avoid heavy frameworks; keep `index.html`, `styles.css`, `app.js`.

## Implementation Constraints

- Must work from `dashboard/index.html` on GitHub Pages.
- Must keep source data path as `data/dashboard-data.json`.
- Must not fabricate numbers.
- Must not require build tooling.
- Must pass:
  - `python3 scripts/generate_dashboard_data.py`
  - `python3 -m py_compile scripts/*.py`
  - `python3 -m pytest -q`
  - `python3 -m json.tool dashboard/data/dashboard-data.json >/dev/null`
  - `node --check dashboard/app.js`

## Tone

Modern and confident, but academically careful. Dashboard should feel like an IoT security operations report, not a hype landing page. Use Indonesian labels/copy where possible.

## Implemented Notes

- Dashboard shell uses static HTML, CSS, and vanilla JS only, with Chart.js and Lucide loaded from CDN.
- Chart panels include DOM fallback bars so the page still exposes values if CDN chart loading fails.
- Metric cards, telemetry, charts, confusion matrix, and evidence sections are populated from `dashboard/data/dashboard-data.json`.
- Copy keeps the required limitations visible: accuracy is not the main claim, normal class is tiny, controlled subsets can overstate performance, SHAP is sampled, split-similarity remains a risk, and global SHAP aggregation differs from Track A-specific interpretation.
